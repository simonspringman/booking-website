Target Architecture

The middleware at wb-api.rwandair.com owns all Amadeus communication. The frontend barely changes.

Browser SPA
(index.html + app.js) → wb-api.rwandair.com
(Middleware / BFF) → Amadeus Digital APIs
(GDS / NDC)
wb-api.rwandair.com → Wallet Service    Amadeus ALLMS
(DreamMiles Loyalty)    Payment Gateways
(Outpace / BK / DPO)    CMS / Admin DB

Frontend (Minimal Changes)

Swap fetch('/api/...') base URL to wb-api.rwandair.com. Keep same request/response shapes. Add auth token headers.

Middleware (Heavy Lift)

Transform Amadeus EDIFACT/JSON responses into our UI contract. Handle session management, caching, error translation, rate limits.

Internal Services (Keep As-Is)

Wallet, CMS, Admin dashboard, chat assistant — stay on your own infrastructure. DreamMiles loyalty migrates to Amadeus ALLMS. Payments via Outpace, Bank of Kigali & DPO.

Phase 5 — Deployment & Go-Live

Weeks 16–18 • Staged rollout with canary deployment and monitoring.

Week 16: Soft Launch

Deploy to production behind feature flag. Internal team + select travel agents only. Monitor Amadeus API response times, error rates, booking success rate.

Week 17: Canary (10% Traffic)

Route 10% of public traffic to new platform. Compare conversion rates with existing booking.rwandair.com. Fix any production-only issues.

Week 18: Full Rollout

100% traffic migration. Old system on standby for 2 weeks. Monitor booking volumes, revenue, support tickets. Post-launch hotfix sprint.

Rollback Plan

  • Instant rollback via DNS/load balancer switch back to current booking.rwandair.com
  • Data sync — any bookings created on new platform are in Amadeus PNR, accessible from both systems
  • Feature flags — individual modules (T2T, wallet payment, check-in) can be toggled off independently

Key Risks & Mitigations

What could delay us, and how we plan to handle it.

# Risk Impact Likelihood Mitigation
1 Amadeus branded fare mapping doesn't match our Lite/Classic/Business model HIGH HIGH Start with a 1-week discovery sprint mapping WB fare families in Amadeus. Build a configurable fare-family engine that can be updated without code changes. Involve revenue management team early.
2 T2T hold durations not supported by Amadeus fare rules (payment deadlines may be shorter) MED HIGH Implement T2T as a "shadow hold" — create the PNR in Amadeus with standard deadline, extend on your side with a background job that re-books if needed. T2T fee covers rebooking risk.
3 Payment gateway delays (Outpace/DPO settlement, BK API timeouts, MTN MoMo callback failures) HIGH MED Implement pending payment queue with automatic retry. Hold PNR until payment confirmed. Add manual resolution queue for stuck payments in admin dashboard.
4 Amadeus API rate limits or downtime during peak booking periods HIGH LOW Cache search results (5-min TTL), implement circuit breaker pattern, show cached fares with "prices may vary" disclaimer during degraded mode.
5 Refund-to-wallet requires two systems to agree (Amadeus void/refund + internal wallet credit) MED MED Use saga pattern: (1) Cancel in Amadeus (2) On success, credit wallet. If step 2 fails, queue for retry. Never credit wallet without confirmed Amadeus cancellation.
6 No Amadeus-experienced developer available in time HIGH MED Engage Amadeus professional services or a certified integration partner for the first 4–6 weeks. Knowledge transfer to internal team during Phase 2.
